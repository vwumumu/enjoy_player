# ADR-0028: Agentic workflow engine choice

## Status

Proposed — pending owner sign-off on the third-party Anthropic-compatible proxy used by every agentic workflow in `.github/workflows/`.

## Context

The repo runs several AI agentic workflows via GitHub Actions:

- `agentic_commands.yml` — dispatcher for `/agentic-*` issue / PR comments
- `.github/workflows/duplicate-code.md`
- `.github/workflows/issue-triage.md`
- `.github/workflows/large-file-simplifier.md`
- `.github/workflows/perf-improver.md`
- `.github/workflows/repo-assist.md`
- `.github/workflows/test-improver.md`
- `.github/workflows/update-docs.md`
- `.github/workflows/agentic-wiki-writer.md`

Each of these runs a model on the full prompt + source tree of the issue / PR / discussion context. They are routed through the upstream `githubnext/agentics` dispatcher (`gh-aw`), which fans out to an Anthropic-compatible chat-completions endpoint.

The dispatcher is configured with a third-party proxy (`https://api.minimaxi.com/anthropic`) that terminates the Anthropic protocol but is not operated by Anthropic, GitHub, or this project's maintainers. As of the issue review that surfaced this ADR (issue #83), no document in `docs/decisions/` records:

1. Who chose the proxy and on what authority
2. What data flows out of GitHub to it (full source tree, issue text, PR diffs, comments)
3. What the data-retention / no-train guarantees are
4. What the operational status of the proxy is, and what the fallback plan is if it goes down

This ADR records the choice and the open questions for owner review.

## Decision

1. **Adopt a vendor-neutral agentic configuration**. The `gh-aw` dispatcher is loaded from a pinned SHA in each workflow (not `main`); the model identifier, API base URL, and any auth headers are set per workflow via the `engine.*` block. Lockfile for the workflow files lives in this repo.

2. **Record the proxy choice here**, including:
   - Proxy endpoint
   - Model identifier
   - Maintainer who signed off
   - Date of last sign-off
   - Known risks (data egress, no-train guarantee, operational continuity)

3. **Default-deny egress**: each workflow that uses the proxy must declare an explicit allow-list assertion in its `permissions:` block and document in its `## Outbound network` section. Default is no internet; the assertion unblocks only the specific endpoints the workflow needs (the proxy base URL plus any docs / pub.dev / api.github.com endpoints it consumes).

4. **Required retention**: prefer providers that publish a zero-retention contract for inference requests. If the chosen proxy cannot confirm zero-retention, mark the workflow `risk: medium` in the dispatcher comments and add an `## Known risk` note to its runbook.

## Consequences

Positive:

- Single source of truth for which engine each workflow uses, with a sign-off trail
- Egress is explicit per workflow — easier to audit and to revoke
- Future engine changes are an ADR revision rather than ad-hoc workflow edits

Negative / open:

- Adds review friction for new agentic workflows (need to update this ADR + add egress allow-list)
- If the proxy is revoked, every workflow that depends on it needs a re-pointed model identifier
- The proxy's data-handling posture must be re-verified at least annually

## Alternatives considered

1. **Self-hosted Anthropic / OpenAI proxy in the maintainer's own infra.** Avoids third-party data egress but adds a maintenance and key-management burden; not picked because the project is maintained by a single small team.
2. **GitHub Models only.** Limited model selection; rate limits are unpredictable for heavy refactor workflows.
3. **Run agents locally via `claude-code` or similar, not via GitHub Actions.** Disconnects the agent from PR / issue context; out of scope for this ADR.

## Implementation notes

- Each `.github/workflows/*.md` file currently picks its own model identifier and proxy URL via `engine:` front-matter. After this ADR lands, the default should be a single shared engine config (likely a workflow-level `config.yml` consumed by all workflows).
- `## Outbound network` section is required on each workflow after this ADR lands.
- The `.lock.yml` companions generated by `gh-aw` are not human-edited and should not be reviewed for ADR compliance — the source workflow is the authoritative version.
